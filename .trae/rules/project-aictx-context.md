# 项目背景 (Context)

## 客户真实诉求与痛点
- **单兵强与组织弱的割裂**：AI IDE (如 Trae, Cursor) 赋予个体极强能力，但团队协作时缺乏组织级上下文规范。
- **上下文污染与幻觉**：不同业务线并行时，AI 缺乏精确的局部上下文，导致跨项目“AI 幻觉”频发。
- **资产腐化与知识传递成本高**：缺乏统一的规则注入机制，AI 生成的代码往往是“一次性”的，文档和代码严重脱节。外包团队和 IT 部门并行多个项目时重构成本极高。

## 商业变现目标 (Quote-to-Cash)
本项目定位为极具潜力的 B2B DevTool，核心是从“IaC”升级为“CaC (Context as Code)”，商业化路径分为三阶段：
1. **开源与标准制定**：开源 aictx CLI，定义“AI 项目规则目录标准”，占领“AI 研发规范化”开发者心智。
2. **规则集市场 (Rule-Hub)**：打造类似 Docker Hub/npm 的 AI 规则与技能分发市场，允许共享垂直领域（如 Web3、SaaS）的优质配置包。
3. **企业级 SaaS 变现 (终局)**：为外包公司或大厂研发效能团队提供私有化的 Meta-Repo 管理平台，实现对企业 AI 宪法和规范同步率的可视化监控（向企业“卖管理铲子”）。

## 一期 MVP 边界声明
- **MVP 核心功能**：实现本地 aictx CLI 工具，跑通“拉取 Meta 仓库规则 -> 动态组装基础与领域规则 -> 平铺注入本地项目”的核心管道逻辑；支持包管理生命周期（如 `postinstall` 钩子）强制同步。
- **一期坚决不做**：暂不开发云端的 Rule-Hub 注册表、不开发可视化 SaaS 管理后台、不涉及强业务代码层面的 AST 解析重构（专注于上下文的组装与下发）。
